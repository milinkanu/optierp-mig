# 09 · Subscription

> Sub‑module of [ACCOUNTING_GAP_AND_PLAN.md](../ACCOUNTING_GAP_AND_PLAN.md). ERPNext workspace card: **Subscription**.

## Status today — 🔴 not built

Zero code. **Net‑new sub‑module.** Build only if there are recurring/contract customers — for an
appliance distributor that usually means **AMC (annual maintenance contracts)**, equipment **rentals**,
or retainer billing. If the business is purely one‑off order→invoice, **defer entirely**.

## Scope decision — recurring invoicing, skip deferred revenue (initially)

The valuable 80% is: **define a plan → attach to a customer → a scheduled job creates the invoice
each cycle.** Deferred‑revenue recognition (booking the income evenly across the contract period
rather than at invoice time) is the advanced 20% — **skip until an accountant asks for it** (master §3.6).

## Build‑out (Phase 4 — on demand)

Data model:
- **Subscription Plan** — **engine master**: `plan_name, item, price, billing_interval (Day/Week/Month/Year), interval_count, currency`.
- **Subscription** — **bespoke document**: `customer, start_date, end_date, status (Active/Past Due/Cancelled/Completed), plans (child: plan + qty), days_until_due, next_invoice_date, generate_invoice_at (Beginning/End of period)`.
- **Subscription Settings** — **engine singleton**: grace period, cancel‑at‑period‑end default.

Logic (bespoke — generates real invoices that post GL):
- A **scheduled job** (`process_subscription`) finds Subscriptions whose `next_invoice_date <= today`,
  generates a Sales Invoice from the plan(s), advances `next_invoice_date`, and updates status.
- Reuse the existing **Sales Invoice** service for the actual GL posting — Subscription just *drives*
  invoice creation; it does not re‑implement posting.
- Handle status transitions (Active → Past Due when unpaid past grace → Cancelled/Completed).

Scheduling:
- Use the same background‑job mechanism the rest of the app uses (confirm what exists — cron/worker).
  The job is idempotent: never double‑bill a period (guard on `next_invoice_date`).

*Acceptance:* a monthly subscription created on the 1st generates one Sales Invoice per month on
schedule, never duplicates, advances correctly, and cancels cleanly without a partial‑period bill
(per the chosen policy).

## Deferred revenue (later, optional)
If/when needed: mark plan items as deferred, and a `process_deferred_accounting` job posts periodic
JEs moving income from a deferred‑revenue liability to recognized income. Out of scope for v1.

## Engine vs bespoke
- **Engine:** Subscription Plan, Subscription Settings (singleton).
- **Bespoke:** Subscription document + the scheduled invoice‑generation job (drives the existing
  Sales Invoice service; no new GL posting code).

## v1 Implementation Plan (designed 2026‑06‑22 — ready to execute)

Grounded in the actual codebase (scheduler = APScheduler `SCHEDULED_JOBS` in `app/core/scheduler.py`;
programmatic invoicing = build a system `CurrentUser({... "company_id": str(cid), "roles":
["System Manager"]})` + `set_company_context(db, cid)` then call the existing `sales_invoice` service —
same pattern `seed_demo.py` uses).

### Data model
- **SubscriptionPlan** — *engine master* (`/m/subscription-plan`, mirrors DunningType):
  `plan_name, item_id (FK items), price (Numeric), billing_interval (Day|Week|Month|Year),
  interval_count (Int, default 1), currency (3), disabled`. (No `name` needed — `naming=field:plan_name`.)
- **Subscription** — *bespoke document* (naming series `ACC-SUB-.YY.-`):
  `name, customer_id, start_date, end_date (nullable), status (Active|Past Due|Cancelled|Completed),
  days_until_due (Int — invoice due_date = posting + this), generate_at (Beginning|End of period),
  next_invoice_date (the engine's cursor), last_invoice_date (nullable, idempotency)`.
  Child **SubscriptionPlanDetail**: `subscription_id, plan_id (FK), qty`.
- **Subscription Settings** — store as a `SystemSetting` JSONB key `subscription_settings` via the
  existing `services/settings.py` (mirrors `print_settings` — **no new table/singleton doctype**):
  `grace_days (int), cancel_at_period_end (bool)`.

### Migration
One migration (next after 0048): `subscription_plans` (engine cols), `subscriptions`,
`subscription_plan_details`. Company‑scoped, **no RLS** (explicit company filter, like the rest).

### Service (`services/subscription.py`) — bespoke
- `create / list / get / cancel` (cancel → status Cancelled; honor cancel_at_period_end later).
- `generate_due_invoice(db, subscription, *, on_date) -> SalesInvoice | None` — the core:
  1. build `SalesInvoiceCreate` items from the plan rows (`item_id`, `qty`, `rate = plan.price`),
     `posting_date = next_invoice_date`, `due_date = posting + days_until_due`.
  2. call `sales_invoice.create_sales_invoice` + `submit_sales_invoice` with a **system actor** scoped
     to `subscription.company_id`.
  3. advance `next_invoice_date` by `interval_count × billing_interval`; set `last_invoice_date`;
     if `end_date` passed → status `Completed`. **All in one transaction** (atomic with the invoice).
- **Idempotency (the critical requirement):** only generate when `next_invoice_date <= on_date`; advancing
  the cursor in the same commit as the invoice means a re‑run can't double‑bill the same period.

### Scheduled job (`app/jobs/subscription.py::process_subscriptions`)
- Add to `SCHEDULED_JOBS`: `{"func": "app.jobs.subscription.process_subscriptions", "trigger": "cron",
  "hour": 2}` (daily 02:00). Runs with **no request context** → opens its own session, queries
  Subscriptions `status IN (Active, Past Due)` AND `next_invoice_date <= today` **across all companies**,
  and calls `generate_due_invoice` for each (catching per‑subscription errors so one bad sub doesn't
  stop the batch — same "skip‑and‑continue" pattern as the dunning/statement batches).

### Endpoints (`api/v1/accounts/subscriptions.py`)
- `POST/GET /subscriptions`, `GET /subscriptions/{id}`, `POST /subscriptions/{id}/cancel`,
  and **`POST /subscriptions/{id}/generate-invoice`** — a manual trigger of `generate_due_invoice`
  (essential so the feature is testable without waiting for cron; also lets a user bill early).
- New permission `"Subscription"` in `seed.py` (Accounts Manager/User).

### Frontend
- Subscription Plan = engine form (`/m/subscription-plan`) — free.
- Subscription = a small bespoke list+form view (`SubscriptionView.vue`) at `/subscriptions`: create
  (customer + plan rows + start/interval), list with status, **"Generate invoice now"** + **Cancel**
  actions, and a link to each generated invoice. Sidebar link under Accounting → (a new "Subscriptions"
  group or under Invoicing).

### Key risks / decisions
- **Month/Year date math**: `Month`/`Year` advancement must clamp month‑end (Jan 31 +1mo → Feb 28).
  Use `dateutil.relativedelta` if it's already a dep, else a small clamped helper (decide at build time).
- **generate_at Beginning vs End**: Beginning → invoice dated at period start (advance cursor by 1 cycle
  after billing); End → invoice dated at period end. v1: default **Beginning** (bill up‑front), expose the field.
- **Past Due**: auto‑flip Active→Past Due when an unpaid generated invoice exceeds `grace_days` — *optional
  for v1* (can derive from invoice outstanding later); ship Active→Completed/Cancelled first.
- **Deferred revenue**: explicitly out of scope (master §3.6).

### Acceptance
A monthly subscription created on the 1st generates exactly one submitted Sales Invoice per month on
its cursor date, advances `next_invoice_date` correctly, never double‑bills on a re‑run (idempotent),
and stops cleanly at `end_date` / on Cancel. Manual `generate-invoice` produces the same result on demand.
